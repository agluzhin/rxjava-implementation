# RxJava Implementation

## О проекте

Данный репозиторий содержит учебную реализацию упрощённой реактивной библиотеки по мотивам RxJava. Проект выполнен на Java 17 и демонстрирует понимание ключевых идей реактивного программирования: работы с потоками событий, цепочек операторов, управления подписками, обработки ошибок и переключения вычислений между потоками.

В отличие от использования готовой библиотеки, здесь реактивная модель реализована вручную. Это позволяет показать не только умение применять RxJava как инструмент, но и понимание того, какие абстракции лежат в её основе: `Observable`, `Observer`, `Emitter`, `Disposable`, `Scheduler`.

Проект ориентирован на учебные цели в рамках курса по многопоточности и реактивному программированию. В коде прослеживается попытка не просто воспроизвести внешний API, а реализовать основные механизмы безопасной доставки событий, корректного завершения потока и управления выполнением в разных пулах потоков.

## Цели реализации

Основные задачи проекта:

- реализовать "холодный" источник данных `Observable`, запускающий генерацию элементов только в момент подписки;
- воспроизвести базовые реактивные операторы `map`, `filter`, `flatMap`;
- реализовать механизм отмены подписки через `Disposable`;
- поддержать переключение контекста выполнения с помощью `subscribeOn` и `observeOn`;
- показать корректную обработку терминальных событий `onError` и `onComplete`;
- подтвердить поведение библиотеки набором модульных и многопоточных тестов.

## Архитектура библиотеки

### 1. `Observable<T>` как центральная абстракция

Класс `Observable<T>` представляет поток данных и является ядром всей библиотеки. Внутри он хранит ссылку на `ObservableOnSubscribe<T> source`, то есть на логику генерации элементов при подписке.

Главная идея реализации:

- `Observable` является "холодным" источником;
- каждая новая подписка вызывает независимый запуск `source.subscribe(emitter)`;
- элементы не генерируются заранее, а появляются только в момент вызова `subscribe(...)`.

Это соответствует классической модели cold observable в RxJava.

### 2. `Observer<T>` как потребитель событий

Интерфейс `Observer<T>` реализует стандартный трёхсигнальный контракт:

- `onNext(T item)` для доставки очередного элемента;
- `onError(Throwable t)` для завершения потока с ошибкой;
- `onComplete()` для успешного завершения.

Тем самым проект следует фундаментальному принципу реактивных стримов: поток может выдать произвольное число `onNext`, после чего завершиться ровно одним терминальным сигналом.

### 3. `Emitter<T>` как канал передачи событий

`Emitter<T>` используется внутри источника и служит посредником между производителем данных и подписчиком. Источник работает не напрямую с `Observer`, а с безопасной прослойкой `Emitter`, которая:

- передаёт элементы подписчику;
- завершает поток;
- сообщает об ошибке;
- умеет проверять, отменена ли подписка через `isDisposed()`.

Такой слой абстракции важен, потому что именно он позволяет централизованно контролировать корректность доставки событий.

### 4. `SafeEmitter` как механизм защиты контракта

Внутри `Observable` используется приватный класс `SafeEmitter<T>`. Это один из самых важных элементов всей реализации.

Его функции:

- не допускает отправку новых элементов после завершения потока;
- гарантирует, что `onError` и `onComplete` вызываются не более одного раза;
- вызывает `dispose()` при терминальном событии;
- блокирует дальнейшую доставку элементов после отмены подписки.

Для этого применяется `AtomicBoolean terminated`, а также объект `SimpleDisposable`, который хранит состояние подписки. Такое решение демонстрирует понимание того, что реактивный контракт должен быть защищён не только логикой тестов, но и самим механизмом отправки событий.

## Поддерживаемые фабричные методы

В библиотеке реализованы основные способы создания потоков:

### `Observable.create(...)`

Позволяет вручную определить логику источника. Это наиболее гибкий способ создания `Observable`, поскольку разработчик полностью контролирует порядок и условия генерации событий.

### `Observable.just(...)`

Создаёт поток из фиксированного набора элементов. Все значения испускаются последовательно, затем вызывается `onComplete()`.

### `Observable.fromIterable(...)`

Создаёт поток на основе коллекции или другого `Iterable`. Этот метод показывает связь реактивной модели с традиционными структурами данных.

### `Observable.empty()`

Сразу завершает поток без передачи элементов.

### `Observable.error(Throwable error)`

Сразу завершает поток ошибкой. Такой метод важен для моделирования аварийных сценариев и для тестирования обработки ошибок.

## Реализованные операторы

### `map(Function<T, R>)`

Оператор преобразует каждый элемент входного потока в новый элемент. Реализация:

- подписывается на исходный поток;
- при каждом `onNext` применяет функцию `mapper`;
- передаёт результат дальше через `emitter.onNext(...)`;
- если `mapper` выбрасывает исключение, оно перенаправляется в `onError`.

Таким образом, проект учитывает, что ошибка может возникать не только в источнике, но и внутри промежуточной стадии обработки.

### `filter(Predicate<T>)`

Оператор пропускает дальше только те элементы, для которых предикат возвращает `true`. Если в предикате возникает исключение, поток завершается через `onError`.

Это соответствует практическому поведению реальных реактивных операторов: ошибка трансформации или фильтрации должна прерывать реактивную цепочку, а не игнорироваться.

### `flatMap(Function<T, Observable<R>>)`

Самый сложный оператор в проекте. Он:

- преобразует каждый элемент исходного потока во вложенный `Observable`;
- подписывается на каждый внутренний поток;
- объединяет все внутренние элементы в единый выходной поток;
- завершает общий поток только после завершения исходного потока и всех внутренних потоков.

Для этого используются:

- `AtomicBoolean sourceCompleted` для отслеживания завершения внешнего потока;
- `AtomicBoolean hasError` для фиксации ошибки;
- `AtomicInteger activeCount` для подсчёта активных внутренних подписок;
- `synchronized (lock)` для защиты конкурентной доставки элементов и терминальных событий.

Это важный архитектурный момент: автор не ограничился "плоской" синхронной обработкой, а попытался воспроизвести логику слияния вложенных потоков и синхронизации общего состояния.

## Управление подписками

### `Disposable`

Интерфейс `Disposable` определяет две операции:

- `dispose()` для отмены подписки;
- `isDisposed()` для проверки текущего состояния.

Этот механизм является ключевым для реактивных библиотек, поскольку асинхронные и потенциально бесконечные потоки должны иметь управляемый жизненный цикл.

### `SimpleDisposable`

Простейшая реализация на базе `AtomicBoolean`. Подходит для индивидуальной подписки.

### `CompositeDisposable`

Позволяет объединять несколько `Disposable` и отменять их одной операцией. Внутри:

- хранится список дочерних подписок;
- используется синхронизация при доступе к коллекции;
- при повторном добавлении после `dispose()` подписка отменяется немедленно.

Это демонстрирует понимание паттерна группового освобождения ресурсов, типичного для UI, сетевых сценариев и сложных подписочных цепочек.

## Модель многопоточности

### Абстракция `Scheduler`

Интерфейс `Scheduler` задаёт два метода:

- `execute(Runnable task)` для выполнения задачи;
- `shutdown()` для освобождения ресурсов.

Идея состоит в отделении логики обработки данных от конкретной стратегии запуска в потоках.

### Реализованные планировщики

#### `IOThreadScheduler`

Построен на `Executors.newCachedThreadPool(...)`.

Назначение:

- I/O-операции;
- задачи с ожиданием сети, диска, базы данных;
- сценарии, где блокировки допустимы и выгодно переиспользование потоков.

Каждому потоку присваивается имя вида `io-thread-*`, что облегчает отладку и тестирование.

#### `ComputationScheduler`

Построен на `Executors.newFixedThreadPool(...)`, где размер пула равен числу доступных процессорных ядер.

Назначение:

- CPU-bound задачи;
- вычисления;
- трансформации данных без длительных блокировок.

Такой выбор показывает понимание разницы между I/O-нагрузкой и вычислительной нагрузкой.

#### `SingleThreadScheduler`

Использует `Executors.newSingleThreadExecutor(...)`.

Назначение:

- последовательное выполнение задач;
- сохранение порядка;
- предотвращение гонок между задачами, которые должны выполняться строго одна за другой.

## `subscribeOn` и `observeOn`

Поддержка этих операторов является сильной стороной проекта, потому что она демонстрирует понимание того, что в реактивном программировании важно различать:

- где выполняется источник данных;
- где выполняется наблюдатель и обработка результатов.

### `subscribeOn(Scheduler scheduler)`

Переносит выполнение логики источника в указанный `Scheduler`. В реализации это достигается через `scheduler.execute(() -> source.subscribe(emitter))`.

Следствие:

- поток-источник может быть перенесён в фоновый исполнитель;
- вызывающий поток не блокируется;
- логика генерации становится асинхронной.

### `observeOn(Scheduler scheduler)`

Перенаправляет доставку событий наблюдателю через указанный `Scheduler`. То есть `onNext`, `onError`, `onComplete` больше не вызываются прямо в потоке источника, а выполняются как задачи в другом исполнителе.

Это важная концепция для UI-приложений, серверной обработки и разграничения этапов пайплайна.

## Вспомогательные блокирующие методы

### `blockingToList()`

Метод подписывается на поток, собирает все элементы в потокобезопасный список и блокирует вызывающий поток с помощью `CountDownLatch`, пока не будет получен `onComplete` или `onError`.

Если поток завершается ошибкой, выбрасывается `RuntimeException` с исходной причиной.

### `blockingToListSafe()`

Обёртка над `blockingToList()`, переводящая `InterruptedException` в `RuntimeException` с восстановлением флага прерывания.

Эти методы полезны не только для удобства, но и как демонстрация мостика между асинхронной моделью и традиционным синхронным тестированием.

## Ограничения учебной версии

Проект осознанно не претендует на полноту промышленной RxJava. В текущей версии отсутствуют:

- backpressure и поддержка `Flowable`;
- богатый набор операторов вроде `zip`, `merge`, `concat`, `retry`, `debounce`;
- централизованное управление ошибками и hooks;
- сложные стратегии диспетчеризации и reentrancy control;
- тонкая оптимизация производительности и минимизация аллокаций;
- полноценный жизненный цикл и автоматическое освобождение scheduler-ресурсов на уровне всего приложения.

Однако для учебной работы это не недостаток, а разумная граница модели: реализован именно тот объём, который позволяет показать ключевые идеи без перегрузки проекта инфраструктурной сложностью.

## Пример использования

```java
Observable.just(1, 2, 3, 4, 5)
        .filter(n -> n % 2 != 0)
        .map(n -> n * n)
        .observeOn(Schedulers.single())
        .subscribe(new Observer<Integer>() {
            @Override
            public void onNext(Integer item) {
                System.out.println("Получено: " + item);
            }

            @Override
            public void onError(Throwable t) {
                System.err.println("Ошибка: " + t.getMessage());
            }

            @Override
            public void onComplete() {
                System.out.println("Поток завершён");
            }
        });
```
